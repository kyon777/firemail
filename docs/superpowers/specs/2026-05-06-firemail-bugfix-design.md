# FireMail 检查邮件与邮件详情弹窗 Bug 修复设计

- 日期：2026-05-06
- 范围：仅修复“检查邮件响应过慢/像没反应”与“查看邮件后关闭按钮失效”两个问题
- 不包含：功能重构、页面改版、Docker 调整、接口协议扩展

## 1. 背景

当前用户反馈两个前端相关问题：

1. 在邮箱列表点击“检查邮件”后，界面响应慢，用户感知为“没反应”。
2. 点击“查看邮件”后，邮件详情弹窗中的关闭按钮失效。

这两个问题都影响基本可用性，因此在后续重构前必须先完成最小修复。

## 2. 已确认现象与证据

### 2.1 检查邮件慢/没反应

现状代码显示：

- `frontend/src/store/emails.js` 的 `checkEmail(emailId)` 走的是 HTTP 接口 `api.emails.check([emailId])`。
- `backend/app.py` 的 `/api/emails/<id>/check` 在提交任务后直接执行 `future.result(timeout=300)`，会同步等待任务完成或超时。
- 同一项目里已经存在 WebSocket 进度通道：
  - 前端：`frontend/src/store/emails.js` 监听 `check_progress`
  - 后端：`backend/ws_server/handler.py` 支持 `check_emails`

这说明当前“单个检查邮件”的路径与“批量检查邮件”的路径不一致：

- 批量检查：异步启动 + WebSocket 通知，用户能立即得到反馈。
- 单个检查：同步阻塞等待，用户点击后长时间没有明确完成前反馈。

### 2.2 查看邮件后关闭按钮失效

现状代码显示：

- `frontend/src/views/EmailsView.vue` 中存在两层邮件相关弹窗：
  - 邮件列表弹窗 `mailListDialogVisible`
  - 邮件内容弹窗 `mailContentDialogVisible`
- 邮件内容内部还会渲染 `EmailContentViewer`，其中又包含附件预览 `el-dialog`
- 这是一个典型的多层 `el-dialog` 嵌套场景，容易出现层级、遮罩、焦点或点击命中问题

用户截图里“关闭”按钮可见但点击无效，优先判断为弹层交互问题，而不是按钮没有绑定。

## 3. 目标

### 3.1 功能目标

1. 点击“检查邮件”后，界面应立即给出明确反馈。
2. 单个邮箱检查过程中，按钮应进入“检查中”状态，并在进度结束后恢复。
3. 查看邮件后，邮件详情弹窗应可被稳定关闭。
4. 不改变现有业务入口和主要页面结构。

### 3.2 非目标

1. 不重构整个邮箱页状态管理。
2. 不把邮件详情整体迁移为新页面或抽屉。
3. 不在本轮统一整理所有历史弹窗实现。

## 4. 修复方案

## 4.1 检查邮件改为异步触发路径

推荐方案：前端单个“检查邮件”与批量检查统一走现有 WebSocket `check_emails` 消息。

### 调整点

#### 前端

文件：`frontend/src/store/emails.js`

- 修改 `checkEmail(emailId)`：
  - 不再优先调用 HTTP `/api/emails/<id>/check`
  - 与 `checkEmails(emailIds)` 对齐，优先复用 WebSocket `check_emails`
  - 若 WebSocket 未连接，再降级走 HTTP 接口
- 在发送检查请求前立即为对应邮箱写入本地 processing 状态，避免用户点击后无感知
- 若后端返回“正在处理中”或连接失败，前端显示明确消息

文件：`frontend/src/views/EmailsView.vue`

- `handleCheck(row)` 保持现有入口，但提示文案改为“已开始检查/正在排队/正在处理中”这一类即时反馈
- 列表按钮依旧通过 `isEmailProcessing()` 切换为 `检查中...`

#### 后端

文件：`backend/app.py`

- 保留 HTTP 接口兼容，但改为“快速返回已启动”而不是阻塞等待完整收信结束
- 启动检查任务后立即返回 `202` 或成功消息
- 仍由既有进度机制更新状态与列表刷新

### 预期效果

- 用户点击按钮后立即看到反馈
- 前端不再被单次 HTTP 调用长时间挂住
- 单个与批量检查共享同一套进度语义，行为更一致

## 4.2 修复邮件详情弹窗关闭失效

推荐方案：保留双层弹窗结构，但显式修正弹层行为，避免遮罩和嵌套弹窗互相干扰。

### 调整点

文件：`frontend/src/views/EmailsView.vue`

- 为邮件内容弹窗增加显式关闭处理函数，如：
  - 关闭时清空 `selectedMail`
  - 将 `mailContentDialogVisible` 置为 `false`
- 为邮件内容弹窗补充更稳定的弹层配置，例如：
  - 需要时使用 `append-to-body`
  - 明确 `destroy-on-close`
  - 明确 `@closed` 或 `@close` 钩子清理状态
- 若当前是从“邮件列表弹窗”中再打开“邮件内容弹窗”，则避免前一个弹窗遮挡后一个弹窗的关闭按钮点击区域

文件：`frontend/src/components/EmailContentViewer.vue`

- 保持组件内附件预览弹窗逻辑，但避免影响父级邮件内容弹窗
- 如有必要，为内部附件预览弹窗也启用独立挂载策略，减少嵌套弹层冲突

### 预期效果

- 邮件内容弹窗的底部关闭按钮和右上角关闭按钮都能生效
- 打开/关闭详情不会残留脏状态
- 附件预览不再干扰父层弹窗操作

## 5. 变更边界

本轮仅允许以下类型变更：

- `frontend/src/store/emails.js`：检查邮件触发逻辑与状态更新
- `frontend/src/views/EmailsView.vue`：弹窗关闭与交互修复
- `frontend/src/components/EmailContentViewer.vue`：必要的嵌套弹窗兼容修复
- `backend/app.py`：单个检查邮件 HTTP 接口快速返回

不做以下改动：

- 替换 Pinia/Vuex 状态模型
- 改造 WebSocket 协议格式
- 大规模样式重写
- 重构邮件详情页面结构

## 6. 验证方案

### 6.1 检查邮件

1. 登录后进入邮箱列表
2. 点击单个邮箱“检查邮件”
3. 预期：
   - 立即出现提示信息
   - 按钮切换为“检查中...”或禁用
   - 后续列表时间/状态可更新
4. 连续点击同一邮箱
   - 预期：不会重复发起多个并发检查
   - 会收到“正在处理中”或等效提示

### 6.2 查看邮件与关闭

1. 点击“查看邮件”打开邮件列表弹窗
2. 再点击某封邮件打开内容弹窗
3. 点击底部关闭按钮
   - 预期：内容弹窗关闭
4. 再次打开内容弹窗，点击右上角关闭
   - 预期：同样关闭
5. 若附件预览存在，再打开附件预览后关闭
   - 预期：不影响父级邮件内容弹窗的关闭

## 7. 风险与控制

### 风险 1：WebSocket 未连接时单个检查失效

控制：

- 保留 HTTP 降级路径
- 明确提示当前连接状态

### 风险 2：修改 HTTP 检查接口后影响旧前端逻辑

控制：

- 返回结构保持兼容，至少保留 success/message 字段
- 由前端根据“已启动”语义处理，而不是要求立即拿到最终结果

### 风险 3：弹窗修复引入新的层级问题

控制：

- 仅对邮件详情相关弹窗加最小配置
- 验证双层弹窗 + 附件预览三层场景

## 8. 实施顺序

1. 修复前端单个检查邮件触发逻辑
2. 修复后端单个检查邮件接口快速返回
3. 修复邮件内容弹窗关闭与清理逻辑
4. 联调验证两个问题是否都稳定复现并消失

## 9. 完成标准

满足以下条件即视为本轮完成：

1. 单个“检查邮件”点击后 1 秒内有明确界面反馈
2. 不再出现前端长时间像无响应的体验
3. 邮件详情弹窗关闭按钮可稳定使用
4. 不引入明显的新交互回归
